# AuthInk Page Patterns

> 本文定义 WeihaoStudio AuthInk 身份认证页面的结构、响应式布局、状态与内容规范。

相关文档：[`DESIGN.md`](./DESIGN.md) · [`COMPONENTS.md`](./COMPONENTS.md) · [`KEYCLOAK_THEME.md`](./KEYCLOAK_THEME.md)

## 1. 页面矩阵

| 页面 | Desktop Light | Mobile Light | Desktop Dark | 关键状态 |
|---|---:|---:|---:|---|
| 登录 | ✓ | ✓ | ✓ | 默认、加载、错误、锁定 |
| 密码重置 | ✓ | ✓ | ✓ | 输入、已发送、链接失效 |
| 更新密码 | ✓ | ✓ | ✓ | 默认、规则错误、成功 |
| TOTP | ✓ | ✓ | ✓ | 默认、验证码错误、信任设备 |
| 恢复码 | ✓ | ✓ | ✓ | 输入、无效、剩余数量提醒 |
| OAuth 中转 | ✓ | ✓ | ✓ | 重定向、账号绑定、冲突 |
| 信息页 | ✓ | ✓ | ✓ | 成功、警告、错误 |

## 2. 全局页面框架

```text
┌──────────────────────────────────────────────────────────┐
│ Brand Visual / Ink Landscape     Auth Flow / Card        │
│                                                          │
│ Logo + Product Story             Page title              │
│ Atmospheric illustration         Form / Action           │
│                                  Secondary actions        │
│                                                          │
│ Footer                                                   │
└──────────────────────────────────────────────────────────┘
```

### 桌面端

- 最小设计视口：`1280 × 720`
- 推荐评审视口：`1440 × 900`
- 设计稿交付比例：`16:9`，推荐 `3840 × 2160`
- 页面采用真正双栏结构，不将移动卡片简单拉宽
- 品牌视觉区：`52–62%`
- 认证操作区：`38–48%`
- 认证卡片宽度：`420–460px`
- 页面内容最大宽度：`1600px`

### 平板端

- `768–1199px`
- 可使用 `42/58` 双栏或弱化品牌视觉区
- 卡片保持完整，不压缩输入框和 OAuth 文案

### 移动端

- `< 768px`
- 改为单列流式布局
- 页头 Logo、表单、OAuth、Footer 按顺序排列
- 水墨山景作为底部或背景淡层，不与表单竞争
- 页面水平安全边距：`16px`
- 卡片最大宽度：`420px`

## 3. Login Page

### 3.1 Desktop Light

布局：

- 左侧：Logo、水墨月山、星辰、简短品牌文案
- 右侧：登录卡片
- 顶部右侧：语言切换、主题切换
- 底部：版权与政策链接

内容顺序：

1. Logo
2. `欢迎回来`
3. `使用您的账号登录`
4. 用户名或邮箱
5. 密码
6. 记住我 / 忘记密码
7. 登录
8. OAuth 分隔器
9. 使用 Google 登录
10. 使用 GitHub 登录
11. 联系管理员（可选）

### 3.2 Mobile Light

移动端不是桌面缩放版：

- Logo 缩小为横向组合
- 页面标题与 Logo 间距减少
- 主按钮和 OAuth 按钮统一 `48px` 高
- OAuth 保持纵向堆叠
- 背景图降低对比度或移至页面底部
- Footer 分行显示

### 3.3 Desktop Dark

- 夜色水墨山景用于品牌区
- Auth Card 使用深墨半透明表面
- 白色 Logo 与细线标题
- 输入框不使用纯黑底，需与页面背景区分
- 主按钮可采用纸白背景 + 墨黑文字
- 保持对比度和焦点可见性

### 3.4 Login States

#### Loading

- 主按钮内部显示加载
- 表单暂时禁用
- 页面保持布局稳定
- 文案：`正在验证身份，请稍候…`

#### Invalid Credentials

- 页面级 Alert：`用户名或密码错误，请检查后重试。`
- 密码字段清空；用户名字段保留
- 焦点回到密码字段

#### Account Locked

- 明确说明账号已锁定
- 避免暴露安全策略细节
- 提供联系管理员或稍后重试入口

## 4. Password Reset

### 输入页

- 标题：`重置密码`
- 说明：`输入您的邮箱，我们将发送密码重置链接。`
- 字段：邮箱或用户名
- 主操作：`发送重置链接`
- 次级操作：`返回登录`

### 发送成功

- 不确认邮箱是否真实存在，避免账号枚举
- 推荐文案：

> 如果该账号存在，我们已发送密码重置说明，请检查邮箱。

- 提供返回登录入口

### 链接失效

- 说明链接已失效或已使用
- 主操作：重新申请
- 次级操作：返回登录

## 5. Update Password

必须展示密码规则，且在用户输入时实时反馈：

- 最小长度
- 大小写要求
- 数字或符号要求
- 不得与历史密码重复（如后端支持）

页面字段：

1. 新密码
2. 确认新密码
3. 保存密码

成功后进入信息页或自动回到应用。

## 6. TOTP Verification

### 页面内容

1. Logo
2. `二步验证`
3. `请输入身份验证器 App 中的 6 位动态验证码。`
4. 6 位验证码输入组
5. `信任此设备 30 天`
6. `验证`
7. `使用恢复码登录`
8. `返回登录`

### Desktop

- 使用宽屏双栏或宽屏居中布局
- 品牌视觉区仍然存在，但弱于登录页
- 验证码输入组完整展示，不压缩

### Mobile

- 6 个输入格必须适配 `320px` 宽屏幕
- 自动弹出数字键盘
- 支持系统一次性验证码自动填充
- 主按钮固定全宽

### Error

- 错误提示：`验证码错误或已过期，请重新输入。`
- 清空或全选验证码，便于立即重试
- 不移动卡片位置

## 7. Recovery Code

- 标题：`使用恢复码登录`
- 说明恢复码用途与一次性属性
- 单输入框，允许粘贴
- 主操作：`验证恢复码`
- 次级操作：`返回二步验证`

成功后如恢复码数量较少，应显示提醒，但不阻断登录。

## 8. OAuth / Identity Provider

### Provider Buttons

- Google
- GitHub
- Microsoft
- Apple
- Generic OIDC / SAML（按租户配置）

统一文案：`使用 {Provider} 登录`。

### Redirect State

- 显示服务商名称
- 显示加载状态
- 文案：`正在跳转至 Google 进行身份验证…`
- 提供可选取消入口

### Account Linking

当外部身份与现有账号冲突：

- 清楚说明需要确认账号归属
- 不展示敏感账号详情
- 提供返回或重新选择登录方式

## 9. Information and Error Pages

### Success

- 图标 + 明确结果 + 下一步
- 例：`密码已更新`

### Warning

- 解释风险与可选操作
- 避免使用过度警报式视觉

### Error

- 用户可理解的错误说明
- 可恢复操作优先
- 技术错误代码可折叠显示，不作为主文案

## 11. Responsive Breakpoints

```css
/* Mobile */
@media (max-width: 767px) {}

/* Tablet */
@media (min-width: 768px) and (max-width: 1199px) {}

/* Desktop */
@media (min-width: 1200px) {}

/* Wide */
@media (min-width: 1600px) {}
```

组件应优先使用容器查询和流式尺寸；断点只处理结构变化。

## 12. 4K 设计稿交付规范

- 主画布：`3840 × 2160`
- 视口示意：Desktop 使用 `1440 × 900` 或 `1600 × 1000` 等比例嵌入
- 不将低分辨率页面截图直接放大
- Logo 使用 SVG
- 图标使用 SVG
- 背景水墨图至少覆盖目标显示尺寸的 `1.5×`
- 文本、组件必须是可编辑矢量或真实前端渲染，不栅格化重绘
- 页面单独交付，不只交总览板

## 13. Content Tone

- 简洁、可信、不夸张
- 避免技术术语堆叠
- 不使用“失败！！”等情绪化表达
- 不承诺无法验证的安全结果
- 关键安全说明以直接陈述句表达
